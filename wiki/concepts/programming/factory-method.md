# 工厂方法模式

> 工厂方法是一种创建型设计模式，在父类中提供一个创建对象的方法，允许子类决定实例化对象的类型。

## 核心思想

将对象的创建逻辑从客户端代码中分离出来，通过工厂方法代替直接调用构造函数（`new`），子类可以通过重写工厂方法改变创建的对象类型。

```
客户端 ──→ 工厂方法 ──→ 具体产品
                    ↑
            具体创建者（子类重写）
```

## 结构

| 角色 | 职责 |
|------|------|
| **产品（Product）** | 定义创建对象的共同接口 |
| **具体产品（Concrete Product）** | 实现产品接口的不同版本 |
| **创建者（Creator）** | 声明返回产品的工厂方法，包含核心业务逻辑 |
| **具体创建者（Concrete Creator）** | 重写工厂方法，返回不同类型的具体产品 |

## 适用场景

1. **无法预知对象确切类型** — 工厂方法将创建代码与使用代码分离
2. **希望扩展框架内部组件** — 子类重写工厂方法即可替换默认组件
3. **希望复用现有对象** — 工厂方法可返回缓存/池化的对象而非每次新建

## 优缺点

| 优点 | 缺点 |
|------|------|
| 避免创建者与产品间的紧耦合 | 引入大量新子类，代码变复杂 |
| 单一职责原则 — 创建代码集中管理 | |
| 开闭原则 — 引入新产品无需改客户端代码 | |

## 与其他模式的关系

- 工厂方法 → 演化为**抽象工厂**、**原型**、**生成器**（更灵活也更复杂）
- 抽象工厂基于一组工厂方法
- 原型不依赖继承，但需要复杂的初始化
- 工厂方法是**模板方法**的特殊形式
