# Wiki Ingest — 新资料摄入工作流

将一份新资料完整纳入 llm-wiki 知识库。你不是在写一份摘要发给用户看 —— 你是在给一个**会长期演化**的知识库添加节点和连接。

## 执行前先读

本 skill 只管"怎么走流程"，**所有格式规范（frontmatter、wikilink、页面角色定义、tag 词表、目录结构）以项目根 CLAUDE.md 为准**。如果还没读过 CLAUDE.md，先读它 —— 特别是"页面格式规范"、"页面角色定义"、"命名规范"三节。

## 工作流骨架（7 步）

每一步都有明确的输入输出，按顺序执行，不要跳步。多步骤任务按 Karpathy 准则陈述简要计划再开工。

### 步骤 0：获取原始内容（输入为 URL 时）

**原则：`raw/` 必须存放原文，不写翻译、不写总结。** 原文的语言、结构、错别字都保留，因为源摘要页做提炼时要能回溯到原文。

- **GitHub 仓库**：优先用 `gh` 获取源文件，避免网页渲染噪音：
  ```bash
  # README
  gh api repos/{owner}/{repo}/readme --jq '.content' | base64 -d

  # 指定文件
  gh api repos/{owner}/{repo}/contents/{path} --jq '.content' | base64 -d

  # 文件树
  gh api repos/{owner}/{repo}/git/trees/HEAD --jq '.tree[].path'
  ```
- **网页 / 文章**：WebFetch 抓取时，prompt 明确要求"返回原文全文，不要摘要、不要翻译、保留原有标题层级"。
- **写入位置**：根据类型放到 `raw/articles/`、`raw/papers/`、`raw/books/`。文件名用小写连字符。

输入已经是本地 `raw/` 文件时，跳过本步。

### 步骤 1：读取源文档

```bash
cat raw/<path>
```

目的是建立自己的理解，不是给用户复述。读完能回答：作者在讲什么？证据是什么？哪些是新观点、哪些是常识？

### 步骤 2：分析并与用户讨论

识别四类信息，列成草稿和用户对齐再落盘：

- **实体**：人物、组织、产品、地点
- **概念**：术语、理论、方法
- **核心观点**：作者的主张和论据
- **新颖性**：相对 wiki 已有内容，哪些是新的、哪些是补充、哪些可能矛盾

**为什么要对齐**：一份资料最终产出 1 个源摘要 + N 个实体 + M 个概念页。这些页面的命名和颗粒度决定了 wiki 未来的连接形态，应该让用户参与决策而不是静默选择。

### 步骤 3：创建源摘要页

位置：`wiki/sources/<subtype>/<slug>.md`（subtype 如 `articles`、`papers`、`morning-briefs`）。

源摘要页是"冷冻的"——忠实提取原文，不做跨源综合。包含：

- 核心论点（3–5 句话）
- 关键摘录（金句、数据、案例，带引用标记）
- 提到的实体（wikilink 到 `entities/`）
- 提到的概念（wikilink 到 `concepts/`）

Frontmatter 必含字段加上 source 特有的 `author` / `url` / `date`，详见 CLAUDE.md。`sources` 字段用 wikilink 格式：

```yaml
sources:
  - "[[raw/articles/example.md|原文]]"
```

### 步骤 4：创建 / 更新实体页

位置：`wiki/entities/<persons|organizations|products|locations>/<slug>.md`

对每个关键实体：
- **已存在**：追加新信息（做了什么、说了什么），在"出现在哪些源摘要中"列表里加上本次源摘要的 wikilink。**不要重写旧内容**，只增量补充。
- **不存在**：创建事实卡片 —— 一句话介绍 + 与本 wiki 的关联 + 来源列表。

不是每个提到的名字都值得建实体页。标准：**会被多篇资料引用 / 是知识图谱的有效枢纽**。一次性出现的小角色不必建。

### 步骤 5：创建 / 更新概念页

位置：`wiki/concepts/<ai|programming|business>/<slug>.md`

概念页是"活的"——用自己的话综合跨源理解。包含：
- 定义（不依赖单一来源）
- 关键特征 / 要素
- 开放问题（还不清楚的、有争议的）
- 来源列表（哪些源摘要页贡献了信息）

**与源摘要页的边界**：源摘要回答"这篇文章说了什么"；概念页回答"这个概念是什么"。如果写概念页时发现只能照搬某一篇文章的说法，说明这个概念页时机未到 —— 暂缓创建，等有第二篇资料再说。

### 步骤 6：更新 index.md

在 `wiki/index.md` 相应类别下添加新页面条目。规则：

- **只展示有内容的类别**（无内容的类别不出现）
- 列表格式：`- [[页面名]] — 一句话描述`
- 头部统计行：`> N 个概念 · N 个实体 · N 个源摘要 — 共 N 页`

统计要数对。先 `ls wiki/concepts/**/*.md` 之类实打实数一遍，不要靠印象。

### 步骤 7：追加到 log.md

追加（不要覆盖）到 `wiki/log.md`：

```markdown
## [YYYY-MM-DD] ingest | 文档标题
- 类型：article | paper | book | morning-brief
- 来源：raw/path/to/file.md
- 新增页面：sources/..., entities/..., concepts/...
- 备注：有冲突 / 新概念 / 跨引用等值得记下的一句话
```

日期用今天实际日期（必要时 `date +%Y-%m-%d`），不要手写估的。

## 完成后自检

开工前用这份 checklist 写计划，收工前逐条验证：

1. `raw/<file>` 是原文、不是翻译或摘要 → 对比抓取结果和落盘内容
2. 源摘要页 frontmatter 完整、`sources` 用 wikilink 格式
3. 每个新实体 / 概念页都 wikilink 到了源摘要，源摘要也 wikilink 到了它们（双向）
4. `index.md` 新条目已加、统计行数字已更新
5. `log.md` 已追加今日记录

任何一条不过就回去补，不要靠用户发现。

## 常见陷阱

- **把翻译塞进 raw/**：原文是英文就保留英文，`raw/` 不是给人读的，是给未来的你做溯源的。
- **概念页照抄源摘要**：如果发现两页内容高度重合，说明概念页没做提炼。概念页要有"你自己的理解"。
- **实体页滥建**：文章里出现的每个公司名都建一页会让图谱变噪。只建真正会被反复引用的。
- **index 统计漏更**：加了新页但忘改头部统计行，日积月累数字就飘了。
- **跳过"分析讨论"直接写**：用户不参与命名决策，wiki 会长成你一个人的审美，后面扩展很难。
